Feedback & Program Directory Page (feedback.html)

This page lets visitors browse four SDG 9-related programmes in a scrollable card directory, read a summary of past participant ratings and reviews, and submit their own feedback through a validated form. It was the page where I did the most JavaScript work on the site.

HTML Structure

Programme Directory Cards

Each programme is a .program-card inside a horizontally scrollable .programs-scroll-container. I attached data-* attributes to every card so the Sort by dropdown has something to sort against without needing a backend.

html
<div class="program-card" data-popularity="5" data-distance="1.2" data-student-friendly="9">
    <img src="images/feedback-innovation-bootcamp.jpg" class="program-card-img"
        alt="Students working on laptops during a coding and robotics workshop">
    <div class="program-card-body">
        <h3>Innovation Bootcamp</h3>
        <p>Learn coding, AI, robotics, and entrepreneurship through hands-on workshops.</p>
        <button type="button" class="button program-btn">Join Now</button>
    </div>
</div>

Ratings & Reviews

Each review's text is split into two spans, a short preview and a hidden full version, so JavaScript can toggle between them without needing to fetch any extra content.

html
<p class="review-text">
    <span class="review-text-short">Short preview sentence...</span>
    <span class="review-text-full" hidden>Full, longer version of the review...</span>
    <a href="#" class="see-more-link">See more</a>
</p>

Feedback Form

The form uses two fieldset/legend groups (Personal Details, Feedback Details), and includes every input type the brief required: text, email, tel, textarea, select, date, and datetime-local. Every field has an adjacent <span class="error-message"> that JavaScript populates on invalid submission.

CSS Architecture

All page-specific styling lives in feedback.css, kept deliberately separate from style.css so I never risk overriding shared rules. A few techniques I used:

  • Flexbox for the scrollable programme row (display: flex; overflow-x: auto;) with scroll-snap-type: x mandatory; so cards snap neatly into place while scrolling.
  • A linear-gradient banner and a dark navy gradient on the ratings summary card to create visual hierarchy without introducing new colours outside the site's palette (#fd6925 orange, #1e2638 navy).
  • A custom ::-webkit-scrollbar styled in the brand orange, so the horizontal scroll affordance doesn't look like a default browser scrollbar.
  • fieldset:focus-within to add a soft glowing border the moment a user starts typing inside a group of fields, giving a live sense of where they are in the form.
  • Three responsive breakpoints (1150px, 800px, 500px) so the layout collapses gracefully from desktop down to mobile.

JavaScript Functionality

All behaviour for this page lives in feedback.js, wrapped in a single DOMContentLoaded listener:

  • Sort by dropdown: reads the selected option and re-sorts the .program-card elements by their data-popularity, data-distance, or data-student-friendly value, then re-appends them in the new order and smooth-scrolls the container back to the start.
  • See more / See less toggle: on click, swaps which of the two review text spans is visible and updates the link's label, using event.preventDefault() so the href="#" anchor doesn't jump the page.
  • Live character counter: listens for input events on the comment textarea and updates the remaining character count against its maxlength.
  • Form validation: on submit, checks every field with a regular expression or length check, writes a specific message into that field's .error-message span, and only shows the confirmation message once every field passes.

Core Rationale

I built the "See more" toggle and the sort function even though the coursework brief says the dropdown didn't need to be functional, because I wanted a genuine demonstration of DOM manipulation beyond the minimum requirement. Splitting review text into two spans instead of truncating with CSS (-webkit-line-clamp) was a deliberate choice: it keeps full control over exactly where the preview cuts off, rather than an arbitrary line count.

Accessibility

  • All programme and review images carry descriptive alt text.
  • The scrollable programme row has tabindex="0" and a visible :focus-visible outline, so keyboard users can reach and scroll it.
  • Every form input has an associated <label for="...">, and error messages use role="alert" so screen readers announce them immediately.
  • The confirmation message uses role="status" aria-live="polite" so it's announced without interrupting the user.

Team Page (team.html)

This page introduces all four team members using interactive thumbnail cards. Hovering, or tabbing to, a card reveals the member's full name, role, and a short summary of what they contributed.

HTML Structure

Each team member is a single .team-member block with tabindex="0", containing an image with a name/role overlay, and a .team-details panel that starts collapsed.

html
<div class="team-member" tabindex="0">
    <div class="team-photo-wrapper">
        <img src="images/team-student3.jpg" alt="Portrait of Student 3" class="team-photo">
        <div class="team-name-overlay">
            <h3>Upeksha Liyanage</h3>
            <span class="team-role-badge">Feedback & Team</span>
        </div>
    </div>
    <div class="team-details">
        <h4>Upeksha Liyanage</h4>
        <p class="team-role">Role: Student 3 — Feedback & Program Directory, Team Page</p>
        <p class="team-pages">...</p>
        <p class="team-summary">...</p>
    </div>
</div>

CSS Architecture

All interactivity on this page is deliberately CSS-only, per the coursework's requirement for the Team page:

  • A CSS Grid (grid-template-columns: repeat(4, 1fr)) lays out the four cards, collapsing to 2 columns and then 1 at smaller breakpoints.
  • .team-member:hover .team-photo scales and darkens the portrait using transform: scale() and filter: brightness().
  • The .team-details panel animates open with a max-height transition (0 → 300px), since height: auto can't be transitioned directly in CSS.
  • :focus-within is combined with :hover/:focus so the panel also opens for keyboard users, not just mouse users.

Core Rationale

I chose max-height over display: none/block toggling specifically because I wanted the expand/collapse to animate smoothly. A hard display switch would jump instantly with no transition, which felt less polished for a hover-driven card interaction.

Accessibility

  • Every .team-member has tabindex="0" so it's reachable purely by keyboard, not just by mouse hover.
  • A visible :focus-visible outline in the brand orange confirms which card is currently focused.
  • All portrait images include descriptive alt text identifying the person.

Content Page (content_ST3.html)

My individual content page, "Community Innovation & Industry", makes the case that SDG 9's goals are advanced as much by small community programmes as by large infrastructure projects, then links that argument directly to the four programmes on my Feedback page.

HTML Structure

The page follows the shared .content-page pattern already defined in the team's global style.css: a full-width banner, then an .article-content-container grid holding a sticky .article-content-sidebar and the main .article-content column.

html
<div class="article-content-container">
    <aside class="article-content-sidebar">
        <nav>
            <ul>
                <li><a href="#introduction">Why Grassroots Innovation Matters</a></li>
                ...
            </ul>
        </nav>
    </aside>
    <div class="article-content">
        <section class="content-page-section-card" id="introduction">
            <h2>Why Grassroots Innovation Matters</h2>
            ...
        </section>
    </div>
</div>

Every sidebar link points to a real id on a <section>, so clicking a link genuinely jumps to that part of the page rather than sitting there as a decorative list. I also added a comparison <table> of the four programmes, and a <fieldset> containing a numbered "Getting Started" list.

CSS Architecture

Most of the layout comes from the global CSS classes Student 1 built for content pages (.content-page-banner, .article-content-fig, etc.), which I reused rather than duplicating. My own page-specific CSS, kept inside an embedded <style> block, only adds what the global rules don't cover:

  • A dark overlay and white heading text on the banner, since the global heading style assumes a lighter background image than the one I used.
  • Border and padding rules for the comparison table and the fieldset, since neither element had existing global styling.
  • A fixed, circular "Go to Top" button implemented purely with an <a href="#top"> and position: fixed; — no JavaScript — per the coursework's specific requirement for this button on content pages.

Core Rationale

Reusing Student 1's existing content-page CSS classes instead of writing my own from scratch kept the four content pages visually consistent as a set, which matters for the site's shared visual identity requirement. I only added CSS where a genuine gap existed (tables, fieldsets, the extra Go to Top button).

Accessibility

  • Heading levels are strictly hierarchical: one <h1> in the banner, <h2> for each major section, <h3> for sub-points — no levels are skipped.
  • All in-article images include descriptive alt text.
  • The table header row uses <th> cells so screen readers can announce column headings when navigating the data.
  • Internal navigation links use real, descriptive text (not "click here"), which is clearer for screen reader users navigating by link list.

Challenges and Lessons Learned

Overriding vs. Extending the Global Button Style

When Student 1 updated the global CSS partway through the project, button styling switched from the plain button element selector to a .button class. My Feedback page buttons only had my own .program-btn/.submit-btn classes, so they silently lost all their styling until I caught it and added class="button program-btn" etc. This taught me to always re-check page-specific CSS against the shared stylesheet after any team update, rather than assuming it still lines up.

Making the Sort Dropdown Actually Functional

The brief explicitly says the "Sort by" dropdown on the Feedback page doesn't need to work. I built it anyway, which meant deciding how to represent "popularity" and "distance" for programmes that don't have real registered data yet. I settled on data-* attributes with placeholder values, clearly documented, so the sorting logic is real and demonstrable even though the underlying numbers are illustrative.

Building the Review "See more / See less" Toggle

My first attempt tried to truncate review text with CSS -webkit-line-clamp, but that only works reliably in WebKit-based browsers and gave inconsistent results elsewhere. Switching to two manually-written spans (short and full) toggled by JavaScript gave me full, predictable control over exactly what the "short" preview says, at the cost of writing two versions of each review by hand.

What's Still Left

I still need to replace every placeholder image path (e.g. images/feedback-innovation-bootcamp.jpg) with real, properly licensed images and update their photo credit links, and swap the bracketed team member names on the Team page for the group's actual final headshots.

Compliance

Copyright and Image Sourcing

Every image placeholder on my pages is annotated with a photo credit line pointing to its source (e.g. Unsplash), following the same pattern Student 1 used on the Home and Content pages. Before final submission I will replace the placeholder href="#" credit links with the real photographer and source URLs for whichever images I actually use, in line with the coursework's requirement that all external resources be properly referenced.

Content Suitability

All copy I wrote for the Feedback, Team, and Content pages is original and was written specifically for this project — none of it is copied from the UN SDG website or any other source. The programme descriptions, reviews, and team summaries are illustrative content I authored to fit the coursework's fictional scenario.

No Commercial or Unsolicited Material

None of my pages contain advertising, sponsored content, or unsolicited promotional material. The "programmes" featured are framed as free educational/community initiatives consistent with the site's SDG 9 mission, not as paid products.

User Data in the Feedback Form

The feedback form collects name, email, phone, and address fields for demonstration purposes only. Per the coursework brief, the form does not submit data to any server or third party — event.preventDefault() stops the native submission entirely, and all validation and the confirmation message happen client-side in the browser. No real user data is collected, stored, or transmitted anywhere.

Network and Resource Use

The only external network request my pages make is to Google Fonts (Poppins and Inter), loaded via a standard <link> tag. No other external scripts, trackers, or analytics are used anywhere on my pages, in line with JANET's guidance on responsible use of external resources.
