Home Page (home.html)

This is the layout for the Home Page, the site's landing page. It has an auto-rotating hero slider, four mission-focused summary cards, and an article directory at the bottom that links out to the site's content pages.

HTML Structure

Hero Slider

The hero is a track of slide divs, and each one carries its photo as a CSS background instead of an img tag. I needed it as a background because the slide animation moves things with transform, and that's easier to do with backgrounds than swapping image sources.

html
<div class="hero-slider-wrapper">
    <div class="hero-slider-track">
        <div class="hero-slide" style="background-image: url(images/hero-bg-img1.jpg);" role="img"
            aria-label="https://pixabay.com/photos/people-industry-construction-3286036/"></div>
        ...
    </div>
</div>

Since a CSS background image has no alt text of its own, I added role="img" and aria-label to each slide so screen readers still get something meaningful instead of nothing. I also updated the slider so it loops properly now: the script clones the first slide and sticks it on the end of the track, so once the slideshow reaches that clone it can snap back to slide 0 with the transition turned off for a moment. That way it loops endlessly instead of jumping back visibly.

Mission & Goal Cards

Four cards summarise the site's main themes. Each one has a credited photo, a short write-up, and a small list of outcomes or actions underneath.

html
<div class="card">
    <h2>The Crisis of Legacy Infrastructure</h2>
    <div class="mgp-card-content-container">
        <div class="mgp-card-img">
            <figure>
                <img src="images/mgp-card-img1.jpg" alt="Explore Dubai's intricate roadways...">
                <figcaption><a class="generic-link" href="...">Photo by Kate Trysh - Pexels</a></figcaption>
            </figure>
        </div>
        <div class="mgp-description">
            <p>Modern societies are heavily reliant on systems engineered in the previous century...</p>
        </div>
    </div>
    <div class="mgp-card-highlights">
        <h3>Key Outcomes :</h3>
        <ul>
            <li>40% reduction in regional energy transmission losses.</li>
            ...
        </ul>
    </div>
    <div class="mgp-card-bottom-text">Local communities and civic planners.</div>
</div>

I split the image and the description text into their own wrapper divs (.mgp-card-img and .mgp-description) instead of mixing them together, so I could size and wrap each one on its own. I also moved the highlights block outside the content container rather than nesting it inside, which keeps every card's outcomes list sitting at the same depth no matter how long the description text ends up being. Figure and figcaption are used again here to keep each photo tied to its credit, same as on the content pages.

Article Linking Section

A "Read our articles..." section closes the page off. There's one big featured thumbnail followed by a grid of smaller ones, and they all link out to the article pages.

html
<div class="mini-article-thumbnail clickable-box" data-href="content_ST1.html" role="link" tabindex="0"
    id="st2_article">
    <div class="mini-article-img-container"><img src="images/gallery/card3_img.jpg"></div>
    <div class="article-meta-data">
        <span>Thimithri vidanapathirana</span>
        <span>03 May 2026</span>
    </div>
    <div class="article-content">
        <h2>Integrating Micro-Enterprises Into Modern Financial Value Chains</h2>
        <p>Discover how grassroots businesses bypass traditional banking barriers...</p>
    </div>
    <div class="article-read-more"><span>Read more</span><svg>...</svg></div>
</div>
js

document.addEventListener('click', (event) => {
    const clickableElement = event.target.closest('.clickable-box');

    if (clickableElement) {
        console.log(`Article click event - id: ${clickableElement.id}`);
        window.location.href = clickableElement.dataset.href;
    }
});

Instead of adding a click listener to every single thumbnail, there's one listener on the whole document that checks if whatever was clicked is inside a .clickable-box, then reads its data-href and navigates there. That means new thumbnails just need the class and the attribute, nothing extra in the script. Each thumbnail also has role="link" and tabindex="0" so it's still discoverable and focusable even though it's a div and not a real anchor tag.

CSS Architecture

Hero Slider Track

I used Grid instead of Flexbox for the slider track because each slide needs to take up exactly one full-width column that can be shifted together as a unit.

css
.hero-slider-track {
    display: grid;
    grid-auto-flow: column;
    grid-auto-columns: 100%;
    width: 100%;
    height: 100%;
}

Why Grid? grid-auto-flow: column with grid-auto-columns: 100% lays out as many equal columns as there are slides automatically. Adding a seventh slide, or the cloned one for the loop, doesn't need any CSS changes at all.

Layered Hero Overlays

The gradient trapezium and the frosted glass pane sit on top of the slider using absolute positioning and a shared clip-path.

css
.hero-overlay-1 {
    position: absolute;
    width: 60%;
    height: 100%;
    z-index: 2;
    clip-path: polygon(0% 0%, 75% 0%, 100% 100%, 0% 100%);
    background-image: linear-gradient(to right top, #fd6925, ..., #5e256d);
}
.hero-overlay-2 {
    position: absolute;
    width: 60%;
    height: 100%;
    z-index: 3;
    clip-path: polygon(0% 0%, 75% 0%, 100% 100%, 0% 100%);
    background-color: rgba(255, 255, 255, 0.6);
    backdrop-filter: blur(40px);
}

Both layers use the same clip-path so they line up, and that's what gives the hero its angled, sliced look instead of a plain rectangle. The backdrop-filter: blur(40px) lets the gradient and the photo behind it show through softly while keeping the heading text easy to read.

Mission Card Layout

Cards use Flexbox in column direction, with a small trick to keep the bottom text strip pinned in place no matter how much content is above it.

css
.card {
    display: flex;
    flex-direction: column;
    gap: 10px;
}
.mgp-card-bottom-text {
    margin-top: auto;
    border-top: 1px solid #FD6925;
}

margin-top: auto inside a flex column pushes the bottom text all the way down to the bottom of the card, so every card's footer strip lines up even if the description text is a different length.

Some design/styling choices

The hero gradient (orange fading into purple) is the boldest colour moment on the page, and everywhere else stays fairly neutral so the gradient panel is what draws the eye first. Article thumbnails get a light background hover state and their "Read more" arrow nudges forward on hover, which matches the hover style used on links elsewhere on the site. Padding and gaps are kept consistent across the mission cards and the article grid so they feel like part of the same layout rather than two separate sections stuck together.

Core Rationale

Keeping the hero's slideshow, gradient and glass pane as separate absolutely-positioned layers means I can tweak any one of them without affecting the others. The role and aria-label attributes on the slides, and the role/tabindex on the article thumbnails, bring back the accessibility meaning that plain divs would otherwise lose. Reusing the same card, figure and figcaption patterns, and the same hover feedback style, from the rest of the site keeps the home page feeling consistent with the content pages instead of like a separate design.

Accessibility

Every hero slide and mission card image has descriptive alt text or an equivalent aria-label. The decorative watermark text in the background is marked aria-hidden so it doesn't get read out loud. Article thumbnails can be reached by keyboard thanks to tabindex, and they're announced as links, though right now the click handling only listens for mouse clicks. Adding a keydown handler for Enter and Space would be a good follow-up so the thumbnails can actually be activated by keyboard too, not just focused.

Gallery Page (gallery.html)

This is the layout for the Gallery Page. It shows a row of interactive cards, and clicking any of them opens a full detail popup pulled from a shared data object in gallery_script.js.

HTML Structure

Thumbnail Grid

The gallery markup itself is fairly bare on purpose. Each thumbnail div only carries an id and an inline theme color variable, and everything else inside it starts empty and gets filled in by the script once the page loads.

html
<div class="gallery-thumbnail" style="--theme-color: rgb(191, 238, 149)" data-id="cat-imhi-imhi">
    <div class="gallery-thumbnail-image-container">
        <img src="" alt="" class="gallery-card-img">
    </div>
    <div class="gallery-thumbnail-badge-container">
        <div class="thumbnail-badge"></div>
    </div>
    <div class="gallery-thumbnail-title-container">
        <h3 class="thumbnail-title"></h3>
    </div>
    <div class="gallery-thumbnail-cta-container">
        <div class="thumbnail-cta">open me !!</div>
    </div>
</div>

The data-id on the card is the key that links it to a matching entry in popupData inside the script. On page load the script looks up that entry, then fills in the image source, alt text, badge text and title straight into the empty elements above, so the actual content only has to exist once in the data object instead of being typed out again in the HTML.

Popup Detail Card

This is the part of the page I spent the most time on. Every card's full detail view lives as one big object inside popupData in gallery_script.js, keyed by the same data-id used on the thumbnail. Here is a trimmed down version of one entry.

js
"cat-imhi-imhi": {
    thumbnailImg: "images/gallery/card1_img.jpg",
    cardBadge: 'INFRASTRUCTURE',
    cardTitle: 'Smart Grid Systems',
    bannerImgLocation: "images/gallery/card1_img_HighRes.jpg",
    bannerImgCredit: `Photo by <a href="...">Karsten Würth</a> on <a href="...">Unsplash</a>`,
    bannerImgAlt: "windmill on grass field during golden hour photo",
    bannerTitle: "Resilient Energy",
    contentTitle: "Next-Gen Grid Decoupling & Distribution",
    contentDescription: "Integrating decentralized AI routing across national power grids...",
    metricLabel_1: "EFFICIENCY GAIN",
    metricLabel_1_SVG: `<svg>...</svg>`,
    metricLabel_1_Value: "38.6%",
    metricLabel_2: "SYSTEM LATENCY",
    metricLabel_2_SVG: `<svg>...</svg>`,
    metricLabel_2_Value: "12ms Response"
}

One object holds everything both the thumbnail and the popup need, the small card image, the badge and title text, the bigger banner photo, the long description, the photo credit, and two metric boxes each with their own inline svg icon and a value. Keeping it all in one place means the thumbnail and the popup are never out of sync with each other, since they are both just reading from the same object.

When a card is clicked, the script looks up its matching entry and builds the entire popup as one template literal, then drops it into a single empty overlay div.

js
card.addEventListener('click', () => {
    if (data) {
        const parser = new DOMParser;
        const tempCreditDoc = parser.parseFromString(data.bannerImgCredit, 'text/html');
        const creditLinks = tempCreditDoc.querySelectorAll('a');

        creditLinks.forEach(link => {
            link.classList.add('generic-link', 'popup-banner-image-credit-link');
            link.setAttribute('target', '_blank');
            link.setAttribute('rel', 'noopener noreferrer');
        });

        const processedBannerImgCredit = tempCreditDoc.body.innerHTML;

        popupOverlay.innerHTML = `
        <div class="popup-card">
            <div class="popup-banner">
                <div class="popup-banner-layer1"><img src="${data.bannerImgLocation}" alt="${data.bannerImgAlt}"></div>
                <div class="popup-banner-layer2"><h1>${data.bannerTitle}</h1></div>
            </div>
            <div class="popup-content-container">
                <div class="popup-long-title-n-des">
                    <h2>${data.contentTitle}</h2>
                    <p>${data.contentDescription}</p>
                    <p>${processedBannerImgCredit}</p>
                </div>
                <div class="popup-content-metrics">...</div>
            </div>
            <div class="popup-customization-panel">...</div>
        </div>
        `;

        popupOverlay.classList.add('is-active');
        document.body.classList.add('modal-open');
    }
});

The credit string is not just dropped in as it is written in the data object. It gets parsed into a temporary document with DOMParser first, so the script can loop through whatever links happen to be inside it and add the generic-link class plus target="_blank" and rel="noopener noreferrer", no matter how many links a particular credit has. That way I only ever write the credit text once in the data object and never have to remember to add those attributes by hand. Once everything is ready, the whole popup markup gets written into popupOverlay.innerHTML in one go, and adding is-active to the overlay and modal-open to the body is what actually reveals the popup and stops the page underneath from scrolling.

Closing the popup does the reverse, and it is handled by a listener on the overlay itself rather than one on the close button alone.

js
popupOverlay.addEventListener('click', (e) => {
    const clickedOverlay = e.target.id === 'popup-overlay';
    const clickedCloseBtn = e.target.closest('#popup-close-btn');
    if (clickedCloseBtn || clickedOverlay) {
        closePopup();
    }
    ...
});

function closePopup() {
    popupOverlay.classList.remove('is-active');
    document.body.classList.remove('modal-open');
    popupOverlay.innerHTML = '';
}

Checking e.target.id === 'popup-overlay' is what lets clicking the dark background outside the card close it, while e.target.closest('#popup-close-btn') catches a click on the little close button in the corner, or anything inside it like the svg icon. Either one calls closePopup(), which removes the classes that make the overlay visible and clears out its innerHTML so the next card clicked starts from a clean slate instead of leaving old content behind.

Customization Panel

At the bottom of the popup there is a small panel with buttons for switching between light and dark mode, changing the font, and toggling a more spaced out readability mode.

html
<div class="switches border-radius-9999px" id="dark-mode-light-mode">
    <button class="dark-light-mode" data-id="custom-panel-btn" data-name="theme" data-value="light-mode" id="is-active-button">...</button>
    <button class="dark-light-mode" data-id="custom-panel-btn" data-name="theme" data-value="dark-mode">...</button>
</div>
<div class="switches" id="font-change">
    <button class="font-change-button" id="is-active-button" data-id="custom-panel-btn" data-name="font" data-value="font-default">Default</button>
    <button class="font-change-button" data-id="custom-panel-btn" data-name="font" data-value="font-mono">Mono</button>
    <button class="font-change-button" data-id="custom-panel-btn" data-name="font" data-value="font-comic">Comic</button>
</div>

Every option button carries the same data-id="custom-panel-btn", so the same click listener on the popup overlay catches every one of them, then reads data-name and data-value off whichever button was actually clicked to figure out what to do.

js
if (name === "theme") {
    if (!(customizeBtn.id === "is-active-button")) {
        const allThemeBtns = popupOverlay.querySelectorAll('[data-name="theme"]');
        const contentContainerElement = popupOverlay.querySelector('.popup-content-container');
        const customizePanelElement = popupOverlay.querySelector('.popup-customization-panel');

        allThemeBtns.forEach(btn => btn.id = "");
        customizeBtn.id = "is-active-button";

        if (value === "light-mode") {
            contentContainerElement.classList.remove('is-dark-mode');
            customizePanelElement.classList.remove('is-dark-mode');
        } else if (value === "dark-mode") {
            contentContainerElement.classList.add('is-dark-mode');
            customizePanelElement.classList.add('is-dark-mode');
        }
    }
}

Clicking a theme button first clears the is-active-button id off both theme buttons, then puts it back on whichever one was just clicked. That id is just being used as a marker for the currently selected option, not for anything to do with real element ids elsewhere on the page. Depending on which value was picked, is-dark-mode then gets added or removed on both the content container and the customization panel, which is what triggers all the dark mode colors defined back in the CSS.

js
} else if (name === "font") {
    if (!(customizeBtn.id === "is-active-button")) {
        const allFontBtns = popupOverlay.querySelectorAll('[data-name="font"]');
        const longTitleDescriptionElement = popupOverlay.querySelector('.popup-long-title-n-des');
        let fontClassToBeRemoved;

        allFontBtns.forEach(btn => {
            if (btn.id === "is-active-button") {
                btn.id = "";
                fontClassToBeRemoved = btn.dataset.value;
            }
        });

        customizeBtn.id = "is-active-button";
        longTitleDescriptionElement.classList.remove(fontClassToBeRemoved);
        longTitleDescriptionElement.classList.add(customizeBtn.dataset.value);
    }
}

Font switching needed a bit more logic than the theme toggle, because there is more than one old class that might need removing depending on what was picked last time. Before adding the new font class, the script first loops through the font buttons to find whichever one currently has the active id, reads its data-value to know which class to strip off the description text, then only after that adds the newly picked one.

js
} else if (name === "readability-btn") {
    const longTitleDescriptionElement = popupOverlay.querySelector('.popup-long-title-n-des');

    if (customizeBtn.id === "is-active-button") {
        customizeBtn.id = "";
        longTitleDescriptionElement.classList.remove(customizeBtn.dataset.value);
    } else if (customizeBtn.id === "") {
        customizeBtn.id = "is-active-button";
        longTitleDescriptionElement.classList.add(customizeBtn.dataset.value);
    }
}

The readability toggle only has one button, so it just flips itself on and off each time it is clicked, adding the readability class when it turns on and removing it again when it turns off. It does not need the reset step the theme and font buttons need, since there is nothing else to clear before it.

CSS Architecture

Layered Grid Background

The thumbnail row sits inside a wrapper with two stacked layers, a plain tinted background behind, and the actual row of cards in front.

css
.gallery-thumbnail-container-layer1 {
    position: absolute;
    z-index: 1;
    background-color: rgba(0, 0, 0, 0.05);
    width: 100%;
    height: 100%;
}
.gallery-thumbnail-container-layer2 {
    position: relative;
    z-index: 2;
    display: flex;
    justify-content: space-between;
    width: 1200px;
}

Using flex with justify-content: space-between keeps the four cards evenly spread across the fixed width row without me having to calculate individual margins. The faint background layer behind it just gives the row a bit of visual separation from the rest of the page.

Hover State

Hovering a card lifts it and makes its call to action more prominent, driven by a class the script toggles on mouseenter and mouseleave.

css
.gallery-thumbnail.is-hovered {
    transform: translateY(-10px);
    box-shadow: 0 12px 20px rgba(0, 0, 0, 0.2);
    cursor: pointer;
}
.gallery-thumbnail.is-hovered .gallery-thumbnail-cta-container {
    background-color: var(--theme-color);
    font-weight: bold;
    font-size: 20px;
    text-transform: uppercase;
}

Each card has its own --theme-color variable set inline in the HTML, so this one CSS rule can give every card a different accent color on hover without needing a separate class per card.

Dark Mode and Font Switching

The popup can change its appearance without reloading anything, just by toggling classes on the content container.

css
.popup-content-container.is-dark-mode {
    background-color: #121824;
}
.popup-long-title-n-des.font-mono {
    font-family: 'SFMono-Regular', Consolas, monospace;
}

Keeping the dark mode and font overrides as separate class based rules means the same base popup markup can look completely different just by adding or removing a class, so the script only has to manage classList changes instead of touching any styles directly.

Core Rationale

Filling in the thumbnail content from a shared data object instead of writing it directly into the HTML means updating a card's photo or wording only has to happen in one place. Splitting the background layer from the card row lets the row sit visually separate from the page without adding extra markup around every single card. The theme color variable and the hover class keep each card's accent color and its call to action tied together with very little CSS.

Accessibility

Images get their alt text filled in from the data object rather than being left blank. That said, the thumbnail cards themselves are plain divs with only a click listener attached, so right now they can't be reached or opened using the keyboard. The same goes for the popup's close button and the overlay background, both only close on a mouse click. Adding tabindex and a keydown handler to the cards, along with an escape key listener for the popup, would be the main things left to fix to make this page properly keyboard accessible.

Content Page (content_ST1.html)

This is the layout for the Content Page (used for the "Integrating Micro-Enterprises" article). It's a shared template (.content-page), so these styles apply site-wide.

Design Inspiration: Based on Odama's work on Dribbble. It uses a full-bleed hero banner with the title layered right on top of the photo instead of a boring text header.

HTML Structure

Hero Banner

Starts with a full-width banner containing the background image, credits, an editor's note link, and the heading overlay.

html
<div class="content-page-banner">
    <img src="images/content-page-banner-img-w2212570.jpg" class="content-page-banner-image"
        alt="Two elderly Vietnamese women weaving traditional bamboo fish traps...">
    <div class="content-page-banner-img-reference">...</div>
    <a class="generic-link content-page-banner-editor-link" href="pageEditor_ST1.html">Editor's page</a>
    <div class="content-page-banner-heading">
        <h1>Integrating Micro-Enterprises Into Modern Financial Value Chains</h1>
        <p>By: Eshan Amarakoon</p>
    </div>
</div>
  • Accessibility: Every image uses a full-sentence alt tag (no filenames or blank strings) so screen readers get the full context.
  • SEO: The <h1> is only used once for the main title to keep the document hierarchy clean.
  • Links: Used actual <a> tags for credits and links so they are keyboard accessible.

Two Column Layout

The main body splits into a sticky sidebar navigation and the actual article content.

html
<div class="article-content-container">
    <aside class="article-content-sidebar">
        <nav>
            <ul>
                <li><a href="#microfinance">Supporting Microfinance...</a></li>
                ...
            </ul>
        </nav>
    </aside>
    <div class="article-content">
        <section class="content-page-section-card" id="microfinance">
            <h2>Supporting Microfinance and Community-Led Credit</h2>
            ...
        </section>
        ...
    </div>
</div>
  • <aside> and <nav> Marks the sidebar as supplementary and tells search engines/screen readers it's an in-page table of contents.
  • Each <section> has a unique id matching the sidebar links for smooth-scroll navigation.
  • <h2> tags Keeps sections neatly organized under the main <h1>.

Other content Elements

A few specific tags were repurposed to handle tables, lists, and images cleanly.

HTML

  • <table> Used strictly for actual data comparisons. Includes <th> tags for accessibility.
  • <fieldset> and <legend>: Borrowed these form tags because they give us a nice, built-in bordered box with an inline title.
  • <figure>: Tethers the image and its source credit together as a single unit.

Interactive Poll

Design Inspiration: The embedded poll was inspired by the reader polls IGN runs at the bottom of their articles — a quick, opinion-style question with a live vote count that reveals each option as a percentage bar once you've voted.

html
<section class="poll">
    <div class="poll-container">
        <div class="poll-header">
            <span id="poll-live-btn">live</span>
            <h2>When purchasing daily essentials or services...</h2>
        </div>
        <div class="poll-body" id="poll-body-form">
            <form id="dummy-article-poll" action="..." method="get">
                <ul>
                    <li>
                        <input type="radio" id="convenience" name="corporate-over-local"
                            value="convenience" required>
                        <label for="convenience">Convenience and speed...</label>
                    </li>
                    ...
                </ul>
            </form>
        </div>
        <div class="poll-body" id="poll-body-result" style="display: none;"></div>
        <div class="poll-footer">...</div>
    </div>
</section>

A small script listens for the form's submit event, tallies the vote against a dummy data object, then swaps #poll-body-form for #poll-body-result and renders each option as a percentage bar — the same reveal-on-vote pattern IGN uses, just built from scratch with vanilla JS instead of a third-party poll widget.

  • Accessibility: Options use real <input type="radio"> / <label for="..."> pairs so the poll is fully keyboard and screen-reader operable, not just clickable <div>s.
  • Progressive enhancement: The <form> still has a real action/method, so it degrades gracefully if JavaScript fails to load.
  • Feedback: Submitting the form disables the button and swaps its label to "Vote Submitted" so users get clear confirmation their vote registered.

CSS Architecture

Article layout

Used Grid for the main container because I needed a fixed width center column flanked by flexible side gutters.

css
.article-content-container {
    display: grid;
    grid-template-columns: minmax(290px, 1fr) 900px 1fr;
    gap: 40px;
    margin: 0 auto;
    margin-top: 50px;
}

Why Grid? It lets us pin the middle reading column exactly to 900px in one line of code. Flexbox would require too much guessing with flex-grow and flex-basis.

Hero Banner Overlay

To get the text floating over the image, the container is relative and the text block is absolute.

css
.content-page-banner {
    height: 60vh;
    position: relative;
}
.content-page-banner-heading {
    position: absolute;
    top: 50%;
    left: 5%;
    z-index: 2;
    background-color: rgba(255, 255, 255, 0.1);
    backdrop-filter: blur(10px);
}

backdrop-filter: blur() with background-color: rgba(255, 255, 255, 0.1); keeps the text perfectly legible over any background image without completely hiding the photo behind a solid box.

Sticky Sidebar

The table of contents stays pinned to the screen while the user reads.

Some design/styling choices

  • Colors are mostly neutral tones (#F4F7F9 background, #475569 text). It's easy and comfortable for eye. And not too fancy
  • All links and buttons have obvious :hover and :active states (like a slight translateX(4px) nudge) so users know they are clickable.
  • Consistent grid gaps (40px) and padding keep the card layout looking unified.

Core Rationale

Used tags like <aside>, <nav>, and <section> instead of <div>. It drastically improves SEO and screen-reader accessibility. The sticky sidebar makes navigating a long article effortless. Frosted glass ensures readability without breaking the visual aesthetic. Descriptive alt text keeps the page screen-reader friendly (no blank names or raw filenames). Using <figcaption> anchors photo credits directly to the image for everyone. Reusing the .content-page-section-card and .card classes everywhere keeps the look unified so it feels like a single, solid design system.

Accessibility

Beyond the semantic tags above: every image (banner, article figures) carries a full-sentence alt description; the heading hierarchy uses exactly one <h1> per page with <h2> for each article sub-section; and every credit/citation link is a real, keyboard-focusable <a> tag rather than plain text.
