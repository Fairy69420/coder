🚀Splash Screen (splash.html)

This is the Splash Screen, the very first page a user lands on. It introduces the site name, the mission statement, and the team, then automatically moves the user on to the Home Page after a short countdown. It doesn't use the shared site template, since the brief says it doesn't need to, but it reuses the same colour palette (navy #1e2638 and orange #FD6925) so the transition into the Home Page still feels like part of the same site.

HTML Structure

Automatic Redirect

The redirect to the Home Page is done with a plain meta refresh tag in the <head>, exactly like the lecture notes on "Automatic Redirect in Links".

html
<meta http-equiv="refresh" content="4;url=home.html">

This alone is enough to send the user to home.html after 4 seconds, with zero JavaScript needed for the redirect itself.

Page Content

The rest of the page is a simple stack of elements inside one wrapper div: an eyebrow subtitle, the site name, a short mission statement, the team member list, a loading animation, the live countdown text, and a skip button.

html
<div class="splash-box">
    <p class="subtitle">SDG 9 - Industry, Innovation and Infrastructure</p>
    <h1>GridWorks</h1>
    <p class="mission">GridWorks is a student project about building smarter...</p>

    <div class="members">
        <span>Student 1</span>
        <span>Student 2</span>
        <span>Student 3</span>
        <span>Student 4</span>
    </div>

    <div class="loader"></div>
    <p id="countdown">Entering site in 4...</p>
    <button id="skipBtn">Skip Intro</button>
</div>

Each team member is just a <span> inside .members, so adding or renaming members later only means editing text, not restructuring anything. The <div class="loader"> is an empty div — all of its visuals come from CSS, not from an image or GIF.

JavaScript Interactivity

Live Countdown

A setInterval ticks once a second and rewrites the countdown text, counting down from 4 to 0.

js
var secondsLeft = 4;
var countdownText = document.getElementById("countdown");

var timer = setInterval(function () {
    secondsLeft = secondsLeft - 1;

    if (secondsLeft > 0) {
        countdownText.innerHTML = "Entering site in " + secondsLeft + "...";
    } else {
        countdownText.innerHTML = "Entering site now...";
        clearInterval(timer);
    }
}, 1000);

The interval clears itself once it hits 0, so it doesn't keep running pointlessly after the countdown finishes and the meta refresh takes over.

Skip Intro Button

Clicking Skip needs to do three things: stop the JS countdown, cancel the meta-refresh redirect so it can't fire again later, and send the user to the Home Page immediately.

js
skipButton.addEventListener("click", function () {
    clearInterval(timer);

    var metaTag = document.querySelector("meta[http-equiv='refresh']");
    if (metaTag) {
        metaTag.remove();
    }

    window.location.href = "home.html";
});

Removing the <meta> tag from the document is what actually cancels the auto-redirect. Without that line, the meta tag would technically still be sitting in the (about to be replaced) document, but removing it is the cleanest way to be sure it can never fire after the user has already chosen to skip.

CSS Architecture

CSS-Only Loader Animation

The loading indicator is a single div styled as a ring with one coloured edge, spun with a @keyframes animation. No GIF or video file is used anywhere.

css
.loader {
    border: 6px solid #333c52;
    border-top: 6px solid #FD6925;
    border-radius: 50%;
    width: 50px;
    height: 50px;
    animation: spin 1s linear infinite;
}

@keyframes spin {
    0% { transform: rotate(0deg); }
    100% { transform: rotate(360deg); }
}

Why this way? Giving the div a border where only the top edge is a different colour, then spinning the whole element, is the simplest possible way to get a spinning loader ring using only CSS — no extra markup, no background images.

Some design/styling choices

  • Dark navy background (#1e2638) with orange (#FD6925) accents was picked to match the same two colours already used for hover states and highlights elsewhere on the site, so the very first screen a visitor sees still feels consistent with the rest of the pages.
  • Team member names are shown as small rounded "pill" tags rather than a plain list, just to break up the wall of centered text a little.
  • Everything is centred and stacked in a single column, since the splash screen is only on screen for 4 seconds — there's no need for a complex layout the user won't have time to read anyway.

Core Rationale

The redirect logic is deliberately split in two: the actual navigation is handled by the meta refresh tag (simple, reliable, works even if JavaScript fails to load), while JavaScript is only responsible for the things that are purely visual or interactive — the live countdown number and the skip button. That way the page still does its core job (getting the user to the Home Page) even in the worst case.

Accessibility

The skip button is a real <button> element, so it's reachable and operable with the keyboard by default. Text contrast is kept high (white/orange text on a dark navy background). One thing I'd still improve: the loader div has no aria-hidden="true" or accessible label, so a screen reader may announce an empty element; adding aria-hidden="true" to it would be a quick fix.

🎯SDG Action Impact Simulator (ais.html)

This page lets users click through six action cards related to SDG 9, and see a live impact score and background change as they select more of them. It uses the shared template (header, main, footer), plus page-specific embedded CSS and an external JavaScript file, ais_script.js.

HTML Structure

Action Cards

Each action card is a plain div holding an emoji, a title, a short description, and its impact point value stored right on the element itself.

html
<div class="action-card" data-points="3">
    <div class="emoji">🏪</div>
    <h3>Support local makers</h3>
    <p>Buy from small local businesses instead of big importers.</p>
    <div class="points">Impact points: 3</div>
</div>

Using a data-points attribute means the JavaScript never has to guess or parse a point value out of the visible text — it can just read data-points straight off whichever card was clicked. All six cards follow this exact same structure, only the emoji, text and point value change.

Results Panel

A single results box sits below the card grid and is what gets updated dynamically.

html
<div id="resultsBox">
    <p>Selected actions: <span id="countText">0</span></p>
    <p>Total impact score: <span id="scoreText">0</span></p>
    <p>Impact level: <span id="levelText">-</span></p>
    <p id="messageText">Click a card above to get started.</p>
    <button id="resetBtn">Reset</button>
</div>

Wrapping every dynamic value in its own <span> with an id means the script only has to update innerHTML on those small spans, rather than rewriting the whole panel's markup every time something changes.

JavaScript Interactivity

Selecting Cards and Adding Up Points

Every card gets a click listener that toggles a selected class, then a shared function loops over whatever is currently selected and adds up the points.

js
for (var i = 0; i < cards.length; i++) {
    cards[i].addEventListener("click", function () {
        this.classList.toggle("selected");
        updateResults();
    });
}

function updateResults() {
    var selectedCards = document.querySelectorAll(".action-card.selected");
    var total = 0;

    for (var i = 0; i < selectedCards.length; i++) {
        var points = Number(selectedCards[i].getAttribute("data-points"));
        total = total + points;
    }

    countText.innerHTML = selectedCards.length;
    scoreText.innerHTML = total;
    ...
}

Recalculating the total from scratch every time (instead of trying to add/subtract a running total on each click) means there's no way for the score to ever get out of sync with which cards are actually selected — it always reflects exactly what's on screen.

Impact Levels and Background Image Changes

Once the total is known, an if/else chain decides which of the three impact levels the user is at, updates the message text, and swaps the page's background image — all done in JavaScript, not with a CSS-only trick.

js
var backgroundImages = {
    low: "url('images/ais/bg-low.svg')",
    medium: "url('images/ais/bg-medium.svg')",
    high: "url('images/ais/bg-high.svg')"
};

if (total === 0) {
    levelText.innerHTML = "-";
    document.body.style.backgroundImage = "none";
} else if (total <= 6) {
    levelText.innerHTML = "Low impact";
    document.body.style.backgroundImage = backgroundImages.low;
} else if (total <= 14) {
    levelText.innerHTML = "Medium impact";
    document.body.style.backgroundImage = backgroundImages.medium;
} else {
    levelText.innerHTML = "High impact";
    document.body.style.backgroundImage = backgroundImages.high;
}

I picked the thresholds myself: 1–6 points is Low, 7–14 is Medium, 15+ is High. They're written directly in the comments above this code in ais_script.js so it's clear where the numbers come from and they can be changed easily.

Reset Button

js
resetBtn.addEventListener("click", function () {
    for (var i = 0; i < cards.length; i++) {
        cards[i].classList.remove("selected");
    }
    updateResults();
});

Reset just clears the selected class off every card, then calls the exact same updateResults() function used everywhere else, so the results panel and background reset back to their empty state through the normal update path instead of a separate special case.

CSS Architecture

Card Grid

css
.card-grid {
    display: flex;
    flex-wrap: wrap;
    justify-content: center;
    gap: 20px;
    max-width: 1000px;
    margin: 30px auto;
}

Why Flexbox? With flex-wrap: wrap, the six cards naturally rearrange themselves into fewer columns on a smaller screen without needing separate media queries.

Selected State

css
.action-card.selected {
    border-color: #FD6925;
    background-color: #fff1e8;
}

Because .selected is only ever toggled by JavaScript, the visual "this card is picked" state and the actual data used for scoring can never disagree with each other — they're driven by the same class.

Core Rationale

Storing each card's point value as a data-points attribute right on the HTML element, instead of hardcoding a separate list of values in the JavaScript, means the cards and their scoring logic can never drift out of sync — changing a card's impact points only ever needs one edit, in one place.

Accessibility

Every image (the small emoji icons are plain text so they're read out naturally by screen readers) needs no alt text of its own. That said, the action cards themselves are plain divs with only a click listener, so right now they aren't reachable or selectable using only the keyboard. Adding tabindex="0", role="button", and a keydown listener (for Enter/Space) to each card would be the main accessibility improvement left to make.

📄Content Page (content_ST2.html)

This is my individual Content Page, "Small Actions, Stronger Industry: Repair, Reuse and Local Innovation". It reuses the shared .content-page/.article-content-container classes from the global stylesheet, the same ones the Content Page template is built around, so it looks and behaves consistently with the other content pages on the site.

HTML Structure

Banner

html
<div class="content-page-banner">
    <img src="images/content-page-banner-repair.svg" class="content-page-banner-image"
        alt="Flat illustration of a workshop building with an orange lightbulb...">
    <div class="content-page-banner-img-reference">...</div>
    <a class="generic-link content-page-banner-editor-link" href="pageEditor_ST2.html">Editor's page</a>
    <div class="content-page-banner-heading">
        <h1>Small Actions, Stronger Industry: Repair, Reuse and Local Innovation</h1>
        <p>By: [Your Name]</p>
    </div>
</div>

Only one <h1> is used on the whole page, for the title in the banner, which keeps the document's heading hierarchy clean for accessibility and SEO.

Sidebar and Sections

Same two-column pattern as the rest of the site's content pages: a sticky sidebar with anchor links, next to the actual article content.

html
<div class="article-content-container">
    <aside class="article-content-sidebar">
        <nav>
            <ul>
                <li><a href="#right-to-repair">Choosing Repair Over Replacement</a></li>
                ...
            </ul>
        </nav>
    </aside>
    <div class="article-content">
        <section class="content-page-section-card" id="right-to-repair">
            <h2>Choosing Repair Over Replacement</h2>
            ...
        </section>
        ...
    </div>
</div>

Each <section> id matches its sidebar link exactly, so clicking a sidebar item smooth-scrolls straight to that section (the smooth scroll itself comes from scroll-behavior: smooth already set on html in the global stylesheet).

Table, Fieldsets and Callout Card

A few different content patterns are used across the six sections to avoid the whole page just being paragraph after paragraph.

html
<!-- comparison table -->
<table>
    <caption><h3>Repair vs. Replace: What Changes</h3></caption>
    <thead><tr><th>Replacing an Item</th><th>Repairing an Item</th></tr></thead>
    <tbody>...</tbody>
</table>

<!-- action checklist -->
<fieldset>
    <legend><h3>A Simple Checklist to Start With</h3></legend>
    <ul>
        <li>Get one broken item repaired instead of replaced this month.</li>
        ...
    </ul>
</fieldset>

<!-- highlighted tip -->
<div class="card">
    <h3>Quick Tip:</h3>
    <p>Before buying something new online, try searching for a local maker...</p>
</div>

Reusing <fieldset>/<legend> for the checklists (even though there's no actual form input inside them) still gives a semantic grouping with a visible caption, and the shared .card class from the global stylesheet means the "Quick Tip" box automatically matches the rounded, bordered look used for cards everywhere else on the site.

Go to Top Button

The brief asks for a Go to Top button built with plain HTML and CSS, not JavaScript.

html
<body id="top">
    ...
    <a href="#top" id="pureGoToTopBtn" class="button">Go to Top</a>
css
#pureGoToTopBtn {
    position: fixed;
    bottom: 20px;
    left: 20px;
    z-index: 50;
}

This is completely separate from the shared #scrollToTopBtn circle in the site template (that one is JavaScript-controlled and only appears after scrolling). This one is just a normal anchor link pointing at id="top" on the <body>, kept on screen at all times using position: fixed, so it works even with JavaScript turned off.

CSS Architecture

Responsive Images

The global stylesheet locks article figures at a fixed 600px width. On small screens that's too wide, so a page-specific media query overrides it just for this page, without touching the global rule.

css
@media (max-width: 700px) {
    .article-content-fig,
    .article-content-fig img {
        width: 100%;
        height: auto;
    }
}

Why a media query here and not in the global stylesheet? Changing the fixed 600px width in the global CSS would affect every content page on the site, not just mine, so keeping the responsive override local to this page's embedded <style> block avoids side effects on other pages.

Core Rationale

Every section links directly back to one of the six actions used in the Action Impact Simulator (repair, e-waste recycling, local makers, student innovation, reporting infrastructure), so the Content Page and the Simulator reinforce the same ideas instead of feeling like two unrelated pages. The table, fieldsets, and callout card are used to break the long-form writing into scannable chunks rather than one continuous wall of text.

Accessibility

Every image, including the banner, has a full-sentence alt description. The heading hierarchy uses exactly one <h1> (the page title) with <h2> for each of the six sections and <h3> for sub-elements like the table caption and fieldset legends. The sidebar is wrapped in <aside> and <nav> so assistive tech can identify it as in-page navigation, and the Go to Top link is a real, keyboard-focusable <a> tag.

🧩Challenges and Lessons Learned

  • Getting the meta-refresh redirect and the JavaScript countdown to agree with each other on the Splash Screen (i.e. making sure Skip Intro reliably cancels the redirect instead of racing against it).
  • Deciding on fair point values and threshold cut-offs for the three impact levels on the Simulator, so that Low/Medium/High actually feel meaningfully different.
  • Fitting a long-form Content Page into the shared sidebar/article layout without the fixed-width images looking broken on a phone-sized screen.

✅Compliance

The University's network connection runs through JANET, the Jisc-operated network used by UK education and research organisations, so anything published on a University server is bound by the JANET Acceptable Use Policy. The main things that policy rules out are offensive or obscene material, content that infringes someone else's copyright, defamatory content, unsolicited commercial material, and anything that misuses the network itself. Here's how the pages I built line up against each of those. Copyright and Image Sourcing Every photo on the Home, Gallery, and Content pages is downloaded and hosted locally rather than hotlinked, and each one is credited with a link back to the photographer and the source site, mostly Unsplash and Pexels, both of which license their photos for free reuse with attribution. The one image I'm not fully confident about is the "bridging the digital divide" graphic on the Home page, which is credited to diversityQ rather than a stock photo site, so its actual licence terms aren't as clear cut as the rest and are worth double-checking before this goes any further. Content Suitability All of the written content across the pages is factual, SDG9-themed commentary, industry statistics, infrastructure topics, and similar, none of it is offensive, obscene, or written about a real person or organisation in a way that could be considered defamatory. No Commercial or Unsolicited Material The site doesn't carry any advertising, doesn't promote a real product or service, and doesn't send anything to anyone unprompted. The only outward-facing links on the pages are the photo credits and the references list. The Interactive Poll and User Data The reader poll on the Content page is the only part of the site that takes user input. Its

tag has action="https://google.com" written into it so the page still degrades sensibly if JavaScript fails to load, but in normal use the submit handler calls event.preventDefault() before that action ever fires, so a vote is tallied against a local dummy dataset in the browser and never actually leaves the page. No names, emails, or any other personal details are collected anywhere on the site. Network and Resource Use All of the interactivity, the slider, the popup, the poll, is vanilla JavaScript that I wrote myself. The pages don't pull in any third-party scripts, trackers, or external APIs, so there's nothing running that could disrupt JANET or another user's connection..
