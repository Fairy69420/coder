User Profile Page

View the live page | View validation evidence

HTML Structure

There is no HTML form on this page at all - the coursework spec for User Profile specifically asks for data collected via prompt() instead. So the <main> only contains a few control buttons, a progress bar and one empty <div class="profileOutput">. Everything that looks like "profile content" is actually built and inserted by JavaScript once the user starts answering questions. 

<div class="profile-output" id="profileOutput">
    <!-- Step cards are injected here dynamically by JavaScript -->
</div>

Inside each generated step card, I used a <dl>/<dt>/<dd> pairing rather than a plain list, because each answer is really a name/value pair (e.g. "name" :"Thamiliny") and <dl> is the semantically correct tag for that instead of stretching a <ul> to do the same job.

CSS Techniques

Each step card uses the site's Global .card class for its background, border and padding, combined with my own .profile-step-card class which only add the reveal animation on top - so I am not refining anything the Global CSS already handle. 

.profile-step-card {
    opacity: 0;
    max-height: 0;
    overflow: hidden;
    transition: opacity 0.4s ease, max-height 0.5s ease;
}
.profile-step-card.revealed {
    opacity: 1;
    max-height: 500px;
}

The progress bar is just two nested divs - a grey .profile-progress-track and an orange .profile-progress-fill whose width is set directly by JavaScript as a percentage. This is a simpler pattern than a native <progress> element, but it let me match the orange accent colour used elsewhere on the site.

Why I Made These Choices

Using prompt() instead of a form matches the coursework requirement directly, but it also changes what "user experience" means on this page - since prompt() interrupts one question at a time, I decided users needed a way to skip questions and come back later rather than being forced through all 9 in one sitting. That's why every prompt accepts an empty answer as a "skip", and the 3 "Revisit Step" buttons only re-ask questions that are still missing, instead of repeating everything. I think this makes the page feel more like a realistic profile builder and less like a rigid quiz, which fits better with the site's overall message about individuals engaging with SDG 9 at their own pace.

Sitemap Page

View the live page | View validation evidence

HTML Structure

I built the sitemap using SVG rather than a static image so it could be interactive and stay text-based (which also keeps the file size small). Each page is drawn as a <g> group containing a <rect> box and a <text> label, with a unique id on the group - this is the same technique used for clickable shapes in our SVG lecture notes (the Home Page/Schedule example), rather than wrapping each shape in an <a> tag. 

<g id="nodeHome" class="site-node root">
    <rect x="390" y="90" width="120" height="40" />
    <text x="450" y="115">Home</text>
</g>

The Home node gets an extra root class so I could give it a different fill colour in CSS, marking it clearly as the main page everything else branches from. Straight <line> elements connect Home to every other node to show the site hierarchy.

CSS Techniques

Rather than writing a separate #id:hover rule for all 12 pages, every group shares a .site-node class, so one rule handles the hover feedback for all of them: 

.site-node:hover rect {
    fill: #FD6925;
}
.site-node:hover text {
    font-weight: bold;
}

This gives clear visual feedback that a box is clickable without repeating the same CSS rule 12 times.

Why I Made These Choices

Clicking is handled entirely in JavaScript rather than with real links, since that's the exact pattern shown in our SVG lecture notes: each group's id is picked up with document.getElementById(), and a click listener changes document.location.href to the matching page. I deliberately kept the sitemap built only from techniques covered in our own module notes, so I can explain every part of it confidently if asked in the demonstration, rather than including something I copied without fully understanding it.

Content Page

View the live page | View validation evidence

HTML Structure

My content page uses the shared .content-page template - banner at the top, a sticky .article-content-sidebar table of contents, and the article body split into 5 <section> elements, each with a unique id (e.g. id="digital-divide") that matches the sidebar's anchor links: 

<li><a href="#digital-divide">What is the Digital Divide?</a></li>
...
<section class="content-page-section-card" id="digital-divide">
    <h2>What is the Digital Divide?</h2>
    ...
</section>

For Section 1, I used a plain <table> because the content there is a genuine side-by-side comparison ("Old Thinking" vs "New Thinking"), which is exactly what tables are for. The other sections use plain <ul>/<ol> lists instead, since their content is just a set of related points rather than a real 2-column comparison.

CSS Techniques

My 5 page-specific embedded CSS rules only style things the Global CSS doesn't already cover: the banner heading's dark overlay so the white title text stays readable over any photo, spacing under each <h2>, spacing around each <figure>, and simple border/padding rules for the table and lists. None of these override existing Global CSS rules - they only add rules for elements the Global stylesheet doesn't touch.

Why I Made These Choices

I picked digital infrastructure and internet access as my angle within SDG 9 because it connects naturally to a financial inclusion focus elsewhere on the site (connectivity needs to exist before a financial inclusion tool is actually useful to anyone) while still being clearly distinct as its own topic. I organised it into 5 sections - the problem itself, community-level fixes, device affordability, digital skills, and student advocacy - so it moves from "what's the problem" to "what can actually be done about it", which matches the coursework's instruction to focus on action rather than just describing the SDG. The sticky sidebar nav makes it easy to jump straight to whichever section a reader cares about, rather than forcing them to scroll through the whole page.

Challenges and Lessons Learned

One challenge was deciding how much JavaScript "sophistication" to use on the User Profile page - my first version used array methods like reduce() and filter() to calculate progress, but I realised that went beyond what we'd actually covered in class, so I'd struggle to explain it confidently if asked about it in the demonstration. I rewrote the logic to rely on simpler loops and counters instead, which took longer to write out but meant I could genuinely explain every line. The same thing came up with the Sitemap page - I originally used <a xlink:href>, <title> and tabindex, none of which are in our SVG lecture notes, so I rebuilt it using only the <g> + addEventListener pattern shown in class. The trade-off is that this version doesn't have the keyboard accessibility or screen-reader labelling I'd originally added, which I know costs some marks under the Sitemap's accessibility requirement - but I decided being able to fully explain my own code mattered more than including techniques I couldn't justify if questioned on them.

Sourcing images was another practical challenge - it took a few tries to find photos that were both properly royalty-free (rather than pulled from a general image search with unclear licensing) and actually matched what each section was arguing, rather than just being generically "tech-related".

Accessibility Considerations

Every image across my three pages has descriptive alt text describing what's actually in the photo, rather than a filename or a blank string. On the Content Page, heading levels stay in order (one <h1> for the page title, <h2> for each section), and photo credits use real <a> tags so they're reachable by keyboard rather than plain unclickable text. On the User Profile page, the step cards use <dl>/<dt>/<dd> so the name/value structure of each answer is conveyed properly rather than just visually implied.

The Sitemap page is the weakest of the three on accessibility, as explained above - since I rebuilt it to only use techniques from our lecture notes, it currently has no keyboard focus support or screen-reader labelling on the individual page nodes. If I revisited this page, I'd add tabindex="0" and a short text label per node, since those are simple additions that wouldn't require any technique beyond what's already used elsewhere on the page.

Compliance with JANET Regulations

All the written content on my Content Page is my own original wording, not copied from the UN SDG website or any other source. Every photo I've used comes from Pexels, Pixabay, or Unsplash's free tier, is properly credited with a link back to the photographer and source site, and I avoided anything sourced from a general image search where the licence couldn't be confirmed. None of my pages contain offensive, defamatory, or otherwise inappropriate content, and everything published reflects genuine academic work rather than anything that could misrepresent the university or breach copyright.
